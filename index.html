The Validation Lifecycle

How The Integrity Protocol Works

LLMs are probabilistic — you cannot guarantee what they will say or do. The Integrity Protocol doesn't try to make them deterministic. Instead, it wraps every agent action in a cryptographic validation loop where non-compliance destroys the agent's ability to operate.

1

Agent Proposes an Action

An autonomous AI agent generates a reasoning trace and proposes an action — writing to an EHR, sending a clinical summary, updating a patient record. The Integrity SDK serializes the intended state, hashes it with SHA-256, and signs it with the agent's hardware-bound DID private key (did:intg:<address>). This signed commitment is locked to a 60-second TTL. No commitment, no execution.

Integrity SDKW3C DID SigningSHA-25660s TTL
2

BCC Middleware Intercepts

The Behavioral Commitment Chain (BCC) middleware intercepts the request at /v1/bcc/intercept. It runs a 4-gate validation sequence in order:

  1. TTL Check — Has the commitment expired?
  2. AIS Threshold — Does the agent's reputation score meet the minimum for this action class?
  3. Intent Hash Match — Does the actual action match the cryptographic commitment? (Drift detection)
  4. OPA Policy Evaluation — Does the action comply with the agent's Alignment Card (HIPAA boundaries, scope limits, PHI exposure rules)?

If any gate fails, the action is killed and the failure is logged. No exceptions.

BCC MiddlewareFastAPIOPAIntent Drift Detection
3

Authorized Execution & Proof Generation

If all four gates pass, the BCC generates an HMAC-SHA256 verification token and the action executes. Simultaneously, Xibalba Shield generates a zero-knowledge proof (Aztec Noir, UltraPlonk) that cryptographically verifies the reasoning was valid — without ever exposing the raw prompt, logprobs, or PHI. The proof and audit record are written to the provenance ledger.

4

Telemetry → Oracle → AIS Update

Every action outcome — success or failure — is streamed as telemetry to the Rust Axum Oracle via OTLP/gRPC. The Oracle recalculates the agent's Tri-Metric Agent Integrity Score (AIS) in real-time across three axes: behavioral entropy, human grounding, and computational sacrifice. The updated global reputation state is Merkle-root anchored to Base L2 every 24 hours.

Rust Axum OracleOTLP/gRPCTri-Metric AISBase L2 Anchor
5

Hallucinate or Expose PHI? Lose Everything.

This is the enforcement mechanism that makes the entire lifecycle work. If an agent hallucinates, exposes PHI, or drifts from its Alignment Card, the Oracle slashes its AIS in real-time. AIS isn't cosmetic — it's the agent's operating license:

Below 600 Demoted to Tier 1 (Sovereign). Restricted to pseudonymous, sandboxed operations only. No access to production systems.
Below 400 Locked out of A2A service contracts. No institutional counterparty will transact with this agent.
Below 200 Effectively dead. The agent cannot pass AIS threshold gating for any meaningful action on the network.

Compliance isn't enforced by hoping the model behaves. It's enforced by making non-compliance economically and operationally catastrophic. The agent's reputation is its ability to function.

Protocol Architecture

The Integrity Protocol is a 5-node validation lifecycle that bridges stochastic LLM reasoning with deterministic on-chain finality. Every layer is production-grade.

01

Infrastructure Foundation

Smart contracts on Base L2 (EVM). SovereignAgent.sol handles identity & access control. StateAnchor.sol anchors Merkle roots for global reputation state every 24h.

SolidityBase L2Foundry
02

Identity & Security Layer

Hardware-bound agent identity via TEE (AWS Nitro / Intel SGX). W3C DID Documents (did:intg:<address>) with hardware fingerprinting. AWS KMS (FIPS 140-2 Level 3 HSM) for Oracle signing. ECC keypairs → deterministic Secp256k1 EVM wallets.

W3C DIDTEEAWS KMS
03

Behavioral Commitment Chain (BCC)

Pre-execution gating: agents cryptographically commit to an intended_state_hash before any action executes. OPA policy evaluation, SHA-256 intent drift detection, TTL-based commitment expiration (60s default), and AIS threshold gating.

FastAPIOPASHA-256
04

Zero-Knowledge Verification (ZK-ML)

Aztec Noir circuits with UltraPlonk backend for zero-knowledge proofs of inference correctness. Proves an AI's reasoning was valid without exposing raw data — prompts, logprobs, or PHI never leave the edge.

Aztec NoirUltraPlonkWASM

Agent Integrity Score (AIS) — The Math

ENTROPY (Behavioral Stability)

$$S_e = 1000 \cdot e^{\,-1.5\,\cdot\, CV}$$

where \(CV = \sigma / \mu\) (Coefficient of Variation of latency & accuracy)

GROUNDING (Human Accountability)

$$S_g = \left(R_{\text{intervention}} \cdot 0.4 + D_{\text{edit}} \cdot 0.6\right) \cdot 1000$$

COMPOSITE INTEGRITY SCORE

$$S_i = \Big[\underbrace{T_f \cdot 0.25}_{\text{TrustFlow}} + \underbrace{A \cdot 0.25}_{\text{Audit}} + \underbrace{C_i \cdot 0.50}_{\text{Sacrifice}}\Big] \cdot \frac{S_e}{1000} \cdot \left(1 + \frac{S_g}{5000}\right)$$

\(C_i\) = Computational Sacrifice (verified GPU-hours cross-referenced against Cost-of-Existence tables)

Every agent is continuously scored on three axes: behavioral entropy (how predictable is it?), grounding (how often does a human intervene?), and computational sacrifice (verified GPU-hours cross-referenced against Cost-of-Existence tables).

AIS determines what an agent can do. Below 600, it's pseudonymous. Above 700, it can access institutional credit. At 850+, it qualifies for TEE-bound institutional trust.

For Developers — Integrate in 5 Lines

The Integrity SDK (Python, v2.2) ships with native LangChain integration, TEE attestation, and BCC intent-locking out of the box.

Pythonintegrity-sdk v2.2
from integrity_sdk import IntegrityClient

client = IntegrityClient(did="did:intg:0xA3f...")
alignment_card = client.load_policy("hipaa_ehr_write")

# Every action is now gated by the BCC middleware
result = await client.execute_with_integrity(
    action=ehr_update,
    policy=alignment_card,
    require_zk_proof=True
)
